# ADR 0002: 显式 Slug 声明与内容哈希幂等上传

- 状态:已接受
- 日期:2026-07-20
- 前置:ADR 0001(Artifact 版本化)

## 上下文

ADR 0001 落地时发现一个问题:服务端自动生成的 slug 是 `<title-base>-<id[:8]>`,同一份文档两次上传的 slug 永远不会相同,"同 slug 重传 → 新版本"的规则自然条件下永远触发不到。第一版实现用 title 派生的 slug 前缀做正则匹配来猜测"这是同一份文档",但这本质上是推断用户意图:两篇标题相同的无关文档会被错误合并进同一 series。

同时,自动递增版本号不具备幂等性:网络重试、双击、页面刷新导致的重复上传会制造内容完全相同的重复版本。

## 决策

### 文档身份由用户显式声明,不做推断

上传表单增加可选 `slug` 字段:

- 提供 slug:声明"这是哪份文档"。series(同 collection 下同 slug)存在则在其下出新版本;不存在则以该 slug 开新 series。
- 不提供 slug:维持自动生成,每次上传都是新文档(新 series)。
- slug 校验:`^[a-z0-9]([a-z0-9-]{0,78}[a-z0-9])?$`,不合法返回 400。

删除 title 前缀正则推断。身份是声明出来的,不是猜出来的。

### 幂等由"身份 + 内容哈希"完成,版本号由服务端分配

- 用户不声明版本号。版本号是 series 内的单调序号,服务端分配 `max+1`;若允许用户声明,与既有版本冲突时覆盖(违反不可变)、拒绝、重放三种语义都需额外规则。
- 幂等规则:声明的 series 存在时,比较上传内容的 sha256 与**最新版本**的 sha256(复用现有 `sha256` 列,服务端自算,不信任客户端提供的哈希):
  - 相同 → 幂等重放:不建行,返回现有最新版本的 JSON,HTTP 200。
  - 不同 → 正常插入 version+1,HTTP 201。
- 只与最新版比较,不与历史版本比较:"回滚到旧内容"(内容等于 v1、最新为 v3)会正确产出 v4,而不会被误判为重放。

### metadata 变更不触发新版本(简单版)

内容相同、仅 tags/描述等 metadata 不同的重传,按幂等重放处理,metadata 改动被忽略。版本的核心是内容;后续如有需要,可另加"更新最新版 metadata"的独立入口(不触碰内容不可变约束)。

## 后果

正面:

- 重复上传(重试/双击/刷新)零副作用,客户端可安全重试。
- 版本归属是显式的,消除了标题撞车导致的误合并。
- 用户可声明的稳定 slug 让公开地址可预期,也为后续链接功能(按 slug 引用)铺路。

负面 / 代价:

- "更新文档"要求用户知道并填写原 slug,比无声明上传多一步认知成本;前端在上传弹窗中以提示文案缓解。
- 并发上传同一 slug 的竞态由 `UNIQUE(collection_id, slug, version)` 兜底,失败方收到 409,需自行重试(可接受)。
